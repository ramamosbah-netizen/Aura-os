# Report — Auth v1: enforced access (JWT identity → request context → RBAC/ABAC)

**Date:** 2026-06-24 · **Repo:** `Desktop/aura-os` (local, branch `main`) · **Increment:** the cross-cutting hardening — the access seam every module already calls goes from *wired* to *enforced*. Additive and staged: nothing breaks, enforcement turns on when identity is present.

> Five modules shared one honest gap — `actorId = null → passes through`. Auth v1 closes it: a verified bearer token becomes the request actor, the RBAC/ABAC platform finally bites, and a denial returns a clean **403** instead of a 500.

---

## What was built

**`shared`** (framework-free, reuses the `node:crypto` the webhook signer already uses):
- `identity/jwt.ts` — minimal **HS256 JWT**: `signJwt` / `verifyJwt` (signature + expiry, constant-time compare) + `AuthClaims`. 5 vitest tests. Self-issued today; a hosted-IdP JWKS verifier can implement the same `verify` shape later.
- `identity/access.ts` — `AccessDeniedError` (typed, carries the reason).

**`core`** (the kernel):
- `tenancy/tenant-context.ts` — refactored to **`AsyncLocalStorage`**: per-request isolation via `run(info, fn)`, so concurrent requests never see each other's actor (the singleton holder would race). `get()` falls back to the dev default outside any request.
- `identity/auth.service.ts` — `AuthService`: `contextFromHeader()` verifies a `Bearer` token → `TenantInfo`; `mint()` for dev tokens; `enabled` from `AUTH_JWT_SECRET`. Off when unset (the API always boots — same pattern as the AI/DB seams).
- `identity/access.service.ts` — `assert()` now throws `AccessDeniedError`.

**`apps/api`**:
- `main.ts` — a global middleware verifies the token and binds the request context (`tenant.run(ctx, next)`); `AccessDeniedFilter` maps `AccessDeniedError` → **403** in one place (every module benefits).
- `auth/auth.controller.ts` — `GET /api/auth/status`, `POST /api/auth/dev-token` (gated by `AUTH_ALLOW_DEV_TOKENS`; the stand-in for a future login UI).
- `auth/auth.seeder.ts` — seeds a `dealChainAdmin` role (`crm.*`, `tendering.*`, `contracts.*`, `projects.*`) granted to `u-admin` at the dev tenant, so enforcement is demonstrable both ways.

**No new dependencies** (`node:crypto`, `node:async_hooks`), **no migration** (roles/grants in memory, as today), **no web changes** (the shell keeps working via the keyless pass-through).

## Verified

- `pnpm build` → **9/9**; `pnpm test` → **76/76** (5 new JWT).
- **Live** (API booted twice):
  - **Auth off** (no secret): `GET /auth/status` → `{enabled:false}`; keyless `POST /tendering/tenders` → **200** (pass-through intact).
  - **Auth on** (secret + dev tokens): `{enabled:true}`; granted `u-admin` token → **200**; ungranted `u-nobody` token → **403** (clean, via the filter — not a 500); keyless → **200** (staged pass-through still holds).

## Decisions

- **AsyncLocalStorage, not a singleton.** Per-request actor isolation is mandatory once requests carry identity — the old holder would leak one request's actor into another under concurrency.
- **Typed error → one 403 filter.** `AccessDeniedError` + a single exception filter fixes the logged "denials → 500" gap for *every* module at once.
- **JWT in `shared`, HS256 via `node:crypto`.** No new dependency; consistent with the webhook HMAC. A Supabase/Auth0 token slots into the same `verifyJwt`/`contextFromHeader` seam later.
- **Staged + additive.** Auth off → boots and passes through (dev/CI unaffected). Auth on → tokens honored and enforced, but anonymous requests still pass through. This is deliberate v1 scope.

## Next — auth v2 (the remaining gap)
