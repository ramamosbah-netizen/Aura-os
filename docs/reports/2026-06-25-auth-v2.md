## What was built

**API (`apps/api`, additive):**
- `POST /api/auth/login` — dev credential policy (requires `AUTH_DEV_PASSWORD` when set, else accepts any), mints a token for the user. The stand-in for a hosted-IdP login.
- `AUTH_REQUIRED` enforcement in the bootstrap middleware: when set (and auth is on), anonymous requests get a **401** except a public allowlist (`/api/health`, `/api/auth/login`, `/api/auth/status`). Off by default → the v1 staged pass-through is preserved.

**Web (`apps/web`):**
- `lib/session.ts` — `SESSION_COOKIE` + a no-`next/headers` decode helper (so `proxy.ts` can share the constant).
- `lib/api.ts` — `sessionToken()` / `authHeader()` / `currentUser()`; `getJson` and **all 6 BFF routes** now forward `Authorization: Bearer <token>` from the httpOnly cookie.
- `app/api/auth/login` + `logout` route handlers — set/clear the httpOnly `aura_session` cookie.
- `app/login/page.tsx` — the sign-in screen (rendered bare; the frame short-circuits on `/login`).
- `app/layout.tsx` + `components/app-shell.tsx` — the frame reads the user server-side and shows a user chip + **Sign out**.
- `proxy.ts` — Next 16 Proxy: optimistic redirect of anonymous users to `/login` when `WEB_AUTH_REQUIRED` is set. The doc is explicit that this is an **optimistic check only — real authz stays at the API**.

## The turbo-cache gotcha (worth recording)

Next inlines a proxy's `process.env` reads **at build time**. `WEB_AUTH_REQUIRED` is not a declared turbo input, so `turbo run build` replays the cached (flag-less) build and the proxy never picks up the flag — which made the lockdown redirect look broken. The fix for verification was a genuine build (`pnpm --filter @aura/web exec next build`, bypassing turbo). **To deploy locked-down, build the web with `WEB_AUTH_REQUIRED=true`** (or add it to `turbo.json`'s build `env`). The API flags (`AUTH_JWT_SECRET`, `AUTH_REQUIRED`) are runtime.

## Decisions

- **The API is the security boundary** (runtime-enforced); the proxy is optimistic UX (build-time flag). This matches the Next 16 guidance exactly.
- **No new dependencies** — `node:crypto`/`node:async_hooks` (API) + Next built-ins (web).
- **Purely additive** — defaults preserve open dev/CI: no flags ⇒ API passes through, proxy is a no-op, BFF forwards an empty header.

## Still open / honest

- **Lockdown is staged, not on by default** — production sets `AUTH_REQUIRED` (API) + `WEB_AUTH_REQUIRED` (web build).
- **Dev login is a stand-in** — no real user store/passwords yet; a hosted IdP (Supabase JWKS) slots into the same `verifyJwt`/`authHeader` seam.
- **⚠️ Rotate before real data** — `AUTH_JWT_SECRET` and the chat-pasted Supabase credentials (user action; I can't rotate live creds).
