# Report — T1.7: Finance module (Invoices v1) — closes the operate loop

**Date:** 2026-06-25 · **Repo:** `Desktop/aura-os` (local, branch `main`) · **Increment:** the **pay** step — Finance / supplier Invoices, billing against Procurement's POs. Completes the operate loop **spend → receive → pay** and is the seventh clone of the module template.

> A supplier invoice bills against a **received PO**. It references the PO by id + title snapshot and carries the supplier + project snapshots down — the operate loop closing, still join-free.

---

## What was built

**`modules/finance` — the `@aura/finance` package:** framework-free `Invoice` domain (references `poId`+`poTitle`; carries `supplierName`, `projectId`+`projectName`; status `draft|approved|paid|cancelled`); store port + Postgres/in-memory (filters tenant/status/po/project); `InvoiceService` (owns data, access seam `assert('finance.invoice.create')`, emits `finance.invoice.created` with po + project + supplier in the payload). 4 vitest tests.

**API** (`apps/api`): `FinanceController` (`POST/GET /api/finance/invoices`, list filters `?status`/`?poId`/`?projectId`). Wired into `AppModule`. Migration `0011_finance_invoices.sql` (`aura_finance_invoices`, indexed tenant/status/po/project, RLS). `AuthSeeder`'s `dealChainAdmin` now includes `finance.*`.

**Web** (`apps/web`): **Invoices** added to the **Operate** nav group → `/finance/invoices` page + BFF route (forwards the auth header) + `components/invoice-create.tsx`. The page fetches invoices **and received POs in parallel** (`/api/procurement/purchase-orders?status=received`); the form's dropdown is those received POs, and picking one inherits supplier + project + value.

## Decisions

- **The operate loop is closed.** Procurement (spend) → Inventory (receive) → Finance (pay), each step referencing its predecessor + carrying the project through-line by snapshot. The whole spend lifecycle is traceable from event payloads, no joins.
- **Compose via the received-PO feed.** The invoice form sources options from the Procurement API filtered to `status=received` — the same status-filtered composition used throughout (won tenders, active contracts, issued POs).
- **Template held a seventh time** (4 deal-chain + 3 operate). Adding a bounded context is now a fully mechanical, ~one-pass operation on this kernel.
